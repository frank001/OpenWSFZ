# D-009 — Developer Handoff: K=10 Confirmation Run (pre-merge gate)

**Date:** 2026-06-21
**Raised by:** Architect (post pass-1 sweep; B2 escalation §0 revised recommendation)
**Defect ID:** D-009 — OSD false-positive manufacture in noise
**Type:** Confirmation measurement + candidate production build. This handoff DOES
authorise a shippable binary **iff** the gates below pass. If any gate fails, **stop
and report** — do not merge.

---

## 1. Context

The pass-1 sweep (`qa/rr-study/results/diag-pass1-sweep-2026-06-21/pass1_sweep.md`)
identified `K_MIN_SCORE_PASS2 = 10` as the recommended operating point: it cuts the
S5 output FP rate by 94% (0.675 → 0.042 /slot) while *improving* hard-subset S7
co-channel recovery over the shipped K=1 baseline.

Two reasons the sweep alone is **not** sufficient to merge:

1. Its S7 figures are the **hard P0–P2 subset** (includes the ~0% P2 3-stack),
   measured post-gating — **not** the full 21-part `co_channel_sweep` headline
   (92.1% reference). We must confirm K=10 does not regress the *easier* S7 parts.
2. Its S5 was 120 slots; the merge gate (AD3) wants the widened S5 with a stated
   FP-rate confidence bound.

This run produces the evidence to merge (or to reject K=10).

## 3. Actions

### Action 1 — Make the change

In `src/OpenWSFZ.Ft8/Native/ft8_shim.c`:

```c
#define K_MIN_SCORE_PASS2       10   /* D-009: was 1; pass-1 sweep selected 10
                                        (S5 FP −94%, no S7 headline regression) */
```

Add a `FT8_SHIM_VERSION 20260029` entry to the changelog comment block at the top of
the file, citing the sweep result path. Rebuild **win-x64 Release**.

> Linux/macOS binaries are already stale (MEMORY: pending CI rebuild) — CI rebuilds
> them on push via the staleness gate + decode.c overlay. Commit win-x64 only.

### Action 2 — Gate G-A: full S7 (headline co_channel_sweep)

Run the **full 21-part** S7 at K=5 (routine gate K), the same configuration as the
`d70aad5` reference (co_channel_sweep 92.14%):

```
python harness/run_scenario.py scenarios/s7-compounding.json --run-dir results/d009-k10-confirm-s7
```

**Pass criterion (AD4-equivalent):** `co_channel_sweep ≥ 89%` (no regression > ~3pp
vs the 92.14% reference). Also report per-part rates and flag any part that regresses
> 5pp vs the `d70aad5` baseline — especially the *easier* parts (P0, P16), which the
sweep's P0–P2 subset did not cover.

### Action 3 — Gate G-B: widened S5 (FP rate with confidence bound)

Run the widened S5 noise scenario, **≥ 100 slots**:

```
python harness/run_scenario.py scenarios/s5-noise-wide.json --run-dir results/d009-k10-confirm-s5
```

**Pass criterion (AD3):** FP rate at or near 0, reported as **FP-per-slot with a 95%
upper confidence bound** (rule of three: 0 FP / 120 slots → ≤ ~2.5% upper bound). A
non-zero but low rate is a **report-and-discuss**, not an automatic pass — bring the
number to the architect.

### Action 4 — Standard build/test gates

- `dotnet build OpenWSFZ.slnx -c Release` → 0 errors, 0 warnings
- `dotnet test OpenWSFZ.slnx -c Release` → all pass (struct-size self-test green;
  note no ABI change here, so layout tests should be untouched)
- `check_native_version.py` confirms the committed win-x64 DLL reports 20260029

### Action 5 — Outcome

- **All gates pass** → this is mergeable. Produce the result dirs + a short
  `confirm_summary.md` (G-A and G-B numbers, verdict). Hand to QA for the normal
  `/opsx:verify` → review → merge workflow (NFR-023 sections 1/2/5 are QA's).
- **G-A fails** (S7 headline regresses) → K=10 trades too much sensitivity on the
  easy parts. **Stop, report.** The fallback operating point is K=5 (re-open the
  Captain decision with the regression data).
- **G-B fails** (FP rate materially > 0) → stop, report; the sweep's S5 result did
  not generalise to the widened corpus.
