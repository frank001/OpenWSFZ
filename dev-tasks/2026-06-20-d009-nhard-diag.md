## 1. Context

D-009 R5 shipped `OSD_NHARD_MAX = 60` as a second OSD gate, claiming calibration
against S5 and S7 histograms (changelog, shim 20260028). That claim was false —
no histograms were collected (AC4 was never completed) and no S5 gate run was
executed before commit.

The QA S5 gate run (Phase 1, s5-noise-wide.json) failed within the first few
minutes. Representative Category B false positives observed at shim 20260028:

```
PK6PAE/R KW7EZM JD89
HB0YAZ RL0SUF/R AH09
847NUD/R NB3YMG/R DI42
VB0GTR/P SW4PRE FF00
TE2TGK 6Z1VQS/P BG94
<...> BU5MPO EK76
ZS0BRG/P K13ZGR GK74
TM4ZTE/P WZ2SIF PM50
VV2DAN 4T0FIN KD17
```

*(These are OSD outputs from pure AWGN — coincidental callsign-like strings, not
captured from real transmissions. They are Category B by the arch-review taxonomy:
structurally valid 3-token Standard QSO messages with 6-char base callsigns and
Maidenhead grids within [A–R]. Text filtering cannot touch them without causing
false negatives on genuine traffic of the same form.)*

### Why the nhard gate may be fundamentally weak for this OSD implementation

`osd_decode` (ft8_lib, depth 2) finds codewords by Hamming minimisation over the
received hard decisions. By construction, ALL codewords it finds — genuine or false
positive — are close to the received bit sequence. It is therefore possible that
Category B FP candidates have `nhard` well below 60, overlapping the genuine-decode
distribution and leaving no usable threshold. This hypothesis must be confirmed or
refuted by measurement before any further implementation work.

### Action 1 — Rebuild with NHARD_DIAG enabled

In `native/ft8_lib_build/patched/ft8/decode.c`, the diagnostic guard is already
present at both OSD sites:

```c
#ifdef NHARD_DIAG
    fprintf(stderr, "OSD_NHARD_SITE1 %d corr %.3f norm %.3f\n",
            nhard, osd_corr, osd_norm);
#endif
```

Add `-DNHARD_DIAG` to the compile flags in `rebuild.ps1` (or the CMake invocation),
rebuild win-x64, confirm the diagnostic binary is running (it will emit
`OSD_NHARD_SITE*` lines to stderr on every OSD hit).

**Do not commit this binary.** It is a diagnostic-only build. The committed
win-x64 DLL at shim 20260028 remains unchanged.

### Action 2 — Collect FP nhard distribution (S5 AWGN)

Run OpenWSFZ with the NHARD_DIAG binary against pure AWGN audio (e.g. VB-CABLE
with white noise, or replay one of the existing AWGN fixtures) for a sufficient
number of FT8 cycles to observe at least 20–30 OSD hits. Capture the stderr
`OSD_NHARD_SITE*` lines to a text file.

**Target:** collect nhard values for OSD candidates that pass CRC-14 and produce
the Category B FP messages seen above. The key question is: are their nhard values
clustered near 87 (as the R5 design assumed) or are they low (e.g. ≤ 30)?

Suggested capture method: redirect stderr to a log file when launching the
application under test, or add a temporary file-write to the existing logging path
if native stderr is not visible from the .NET host.

Record:
- nhard value of each OSD hit (from `OSD_NHARD_SITE*` lines)
- Whether it was accepted or rejected at `OSD_NHARD_MAX = 60`
- The corresponding decoded message (correlate by timestamp from ALL.TXT)

## 5. Decisions made and why

**D1 — Diagnostic-only handoff.** The R5 fix shipped without the histogram data
it claimed to be calibrated against. The wrong threshold choice caused a gate
failure. The same error must not be repeated with a revised threshold. The
architect must see the actual distributions before prescribing a gate.

**D2 — No threshold guess.** Given that OSD finds codewords by Hamming
minimisation, the FP nhard distribution may overlap significantly with the genuine
distribution, making any single threshold futile. The architect must determine
whether nhard is a viable discriminant at all for ft8_lib's depth-2 `osd_decode`,
or whether a different approach (e.g. a signed-LLR quality score, OSD order
statistic, or the OSD iteration count at which the valid codeword was found) is
required.

**D3 — Category B is the binding constraint.** All nine observed FPs have 6-char
base callsigns, valid portable suffixes, and valid Maidenhead grids. Text filtering
is exhausted. The only remaining lever is the native OSD gate.
