## 1. Context

### What R2 achieved

The R2 fix (commit `8eea3c4`, shim 20260026) applied two complementary changes:
- **Native OSD correlation gate** at `OSD_CORR_THRESHOLD = 0.10` in `decode.c` (both call
  sites): rejects candidates where the normalised inner product of the decoded codeword and
  the input LLRs falls below the threshold.
- **Text-layer Gap A and Gap C extensions** in `Ft8Decoder.IsPlausibleMessage`: 2-token
  messages now check both tokens for oversize; the CQ early-return was removed so that
  garbage 3-token CQ messages fall through to grid/report validation.

The R2 run (`8eea3c4`, K=1, 12 noise-only slots) produced **9 FP events / 12 slots =
75.0% event FP rate** (threshold ≤ 6.0%).

### Why R2 is insufficient

QA analysis of `qa/rr-study/results/2026-06-20-8eea3c4/S5_matched.csv` identified two
distinct residual FP categories:

**Category A — 4-token anomalies (5 of 9 events):**
```
Q1AAA Q2BBB/R R OH76        ← CALLSIGN CALLSIGN R GRID — not a valid Type 1 message
Q3CCC/P Q4DDD R EQ48        ← same pattern
Q5EEE/R Q6FFF/R R CH00      ← same pattern
Q7GGG/P Q8HHH R EO10        ← same pattern
Q9III/P Q1JJJ O/P QG94      ← CALLSIGN CALLSIGN <garbage> GRID
```

FT8 Type 1 (Standard QSO) is **always exactly 3 tokens**. The only valid 4-token FT8
messages are `CQ <geographic-modifier> <CALLSIGN> <GRID>` (e.g. `CQ DX Q1ABC FN42`).
The current `IsPlausibleMessage` code accepts all 4+ token messages unconditionally.
That is the text-layer gap. It does not require regex — a single `token0 == "CQ"` check
on the 4-token branch is sufficient.

**Category B — structurally valid 3-token messages (3 events):**
```
0N4KV 7F4VSB JK98
8H2LOJ JL0NTD/R JI99
H34INX SZ9DTT/P LB12
```

These have valid callsign lengths, valid portable suffixes, and grids whose letters are
within [A–R]. They are **indistinguishable from real traffic by text alone**. No text
filter — regex or otherwise — can address these. The native gate threshold must be raised.

**Category C — 1 event:**
```
CQ <...>
```

This is a valid FT8 Type 4 form (hashed non-standard callsign CQ). It cannot be
rejected without false negatives. Category C FPs must be eliminated by the native gate.

### Residual after Category A fix

Fixing Category A eliminates 5 events. Residual = 4 events / 12 slots = **33.3% event
FP rate** — still far above the 6.0% threshold. Raising `OSD_CORR_THRESHOLD` is the
primary fix.

### Action 2 — Text-layer 4-token rule in `Ft8Decoder.IsPlausibleMessage`

**File:** `src/OpenWSFZ.Ft8/Ft8Decoder.cs` — `IsPlausibleMessage`

This adds `thirdSpace` to the existing block and handles exactly-4-token messages.

**Locate the existing D9-R3 block** (currently at approximately line 380):

```csharp
        {
            int firstSpace  = text.IndexOf(' ');
            int secondSpace = firstSpace >= 0 ? text.IndexOf(' ', firstSpace + 1) : -1;

            if (firstSpace > 0 && secondSpace < 0)
            {
                // 2-token message: "TOKEN0 TOKEN1"
                // Check both tokens — the CQ/DE/QRZ restriction on token0 was too narrow;
                // patterns like "<...> M5E5B91HFHL" and "9ULLPTCDZH <...>" slipped through.
                // IsCallsignOversized already exempts hash references (<...) and short keywords.
                string token0 = text[..firstSpace];
                string token1 = text[(firstSpace + 1)..];
                if (IsCallsignOversized(token0) || IsCallsignOversized(token1))
                    return false;
            }
            else if (firstSpace > 0 && secondSpace > firstSpace
                     && text.IndexOf(' ', secondSpace + 1) < 0)
            {
                // Exactly 3-token message: "TOKEN0 TOKEN1 TOKEN2"
                string token0 = text[..firstSpace];
                string token1 = text[(firstSpace + 1)..secondSpace];
                if (IsCallsignOversized(token0) || IsCallsignOversized(token1))
                    return false;
            }
            // 4+ token messages (CQ DX, contest, free-text): not validated here.
        }
```

**Replace with:**

```csharp
        {
            int firstSpace  = text.IndexOf(' ');
            int secondSpace = firstSpace  >= 0 ? text.IndexOf(' ', firstSpace  + 1) : -1;
            int thirdSpace  = secondSpace >= 0 ? text.IndexOf(' ', secondSpace + 1) : -1;

            if (firstSpace > 0 && secondSpace < 0)
            {
                // 2-token message: "TOKEN0 TOKEN1"
                // Check both tokens — the CQ/DE/QRZ restriction on token0 was too narrow;
                // patterns like "<...> M5E5B91HFHL" and "9ULLPTCDZH <...>" slipped through.
                // IsCallsignOversized already exempts hash references (<...) and short keywords.
                string token0 = text[..firstSpace];
                string token1 = text[(firstSpace + 1)..];
                if (IsCallsignOversized(token0) || IsCallsignOversized(token1))
                    return false;
            }
            else if (firstSpace > 0 && secondSpace > firstSpace
                     && thirdSpace < 0)
            {
                // Exactly 3-token message: "TOKEN0 TOKEN1 TOKEN2"
                string token0 = text[..firstSpace];
                string token1 = text[(firstSpace + 1)..secondSpace];
                if (IsCallsignOversized(token0) || IsCallsignOversized(token1))
                    return false;
            }
            else if (firstSpace > 0 && secondSpace > firstSpace && thirdSpace > secondSpace
                     && text.IndexOf(' ', thirdSpace + 1) < 0)
            {
                // Exactly 4-token message: "TOKEN0 TOKEN1 TOKEN2 TOKEN3"
                // The only valid 4-token FT8 forms are "CQ <modifier> <callsign> <grid>"
                // (e.g. "CQ DX Q1ABC FN42", "CQ NA Q9XYZ EN37").
                // Patterns like "CALLSIGN CALLSIGN R GRID" are OSD CRC-14 false alarms
                // (D-009 R2 verification, Category A) — they are not Type 1 FT8 messages.
                string token0 = text[..firstSpace];
                if (!token0.Equals("CQ", StringComparison.Ordinal))
                    return false;
            }
            // 5+ token messages: not validated here.
        }
```

**Note on the 3-token condition change:** The old condition was
`text.IndexOf(' ', secondSpace + 1) < 0` (no third space). The new equivalent is
`thirdSpace < 0` — functionally identical, uses the pre-computed value.

## 5. Decisions Made and Why

**Decision 1 — Raise threshold to 0.15, not 0.20, as the first calibration step.**
The R2 handoff specified 0.05 increments. Jumping directly to 0.20 risks over-suppressing
genuine weak-signal decodes in the S7 co-channel scenario. The incremental approach allows
the S7 regression check to catch any over-suppression before it causes a gate failure.

**Decision 2 — 4-token rule: `token0 == "CQ"`, not a full modifier whitelist.**
A whitelist of geographic modifiers (DX, NA, EU, AS, AF, OC, SA, VHF, etc.) would be more
precise but creates a maintenance surface. FT8 is a controlled protocol; any valid 4-token
message starts with CQ by spec. An unknown modifier in position 1 is preferable to a false
negative on a valid but unlisted form. The CQ check alone is sufficient and safe.

**Decision 3 — No regex.**
The Category A FPs are caught by a `token0 == "CQ"` check — no regex is warranted or
useful here. The Category B FPs (structurally valid 3-token messages) cannot be caught
by any text-layer rule, regex or otherwise; they require the native gate to reject them
at the signal-processing level. Regex was assessed and found to offer no advantage over
the current string-manipulation approach for any category in this run.

**Decision 4 — Do not reject `CQ <...>` (Category C).**
`CQ <...>` is a valid Type 4 FT8 message form produced when the operator CQs with a
non-standard callsign not yet in the hash table. Rejecting it would be a false negative
in real operation. It must be suppressed by the native gate, not by text filtering.
