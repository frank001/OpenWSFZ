## 1. Context

### R3 outcome

R3 (commit `2e65a63`, shim 20260027) raised `OSD_CORR_THRESHOLD` from `0.10` to `0.15` and added
the 4-token non-CQ text-layer rule. The developer ran S5 and observed false positives on
noise-only slots. WSJT-X produces zero FPs on the same noise input.

The S5 analysis step was not run — `qa/rr-study/results/2026-06-20-2e65a63/` contains only
`truth.csv`; there is no matched CSV, no FP message list, and no `report.md`. The FP count
at `0.15` is not yet precisely known but FPs were confirmed by the developer.

### Revised target

AC6 was originally set at ≤ 6.0% event FP rate. QA has revised this target to **0.0% event
FP rate**, matching WSJT-X performance on noise-only S5 input. Any observed FP on a noise-only
slot is not acceptable.

### Two blocking defects

1. **S5 FP not eliminated** — `OSD_CORR_THRESHOLD = 0.15` is insufficient. Calibration must
   continue until 0 FP events are observed on S5.

2. **NFR-021 violation** — ten callsigns with valid ITU-allocated prefix formats appear in
   `tests/OpenWSFZ.Ft8.Tests/D009FpFilterTests.cs` (lines 166–170, 239–240) and in the
   committed R3 handoff `dev-tasks/2026-06-20-d009-fp-filter-r3.md` (Section 1, AC3). These
   are `JT6KU`, `EH2AOE`, `WH8EYF`, `LC8WMR`, `K60IFO`, `9O5GEL`, `IU7WEX`, `6N8XOX`,
   `KR9NHK`, `MR1I` — all plausibly assignable to natural persons under their respective
   ITU-allocated prefixes (Mongolia, Spain, American Samoa, Norway, USA, DR Congo, Italy,
   Mexico, England). Per NFR-021 this is a merge-blocking defect regardless of whether the
   callsigns are currently licensed.

## 5. Decisions Made and Why

**Decision 1 — Target: 0.0% S5 FP rate (revised from ≤ 6.0%).**
WSJT-X achieves zero FPs on S5 noise-only input. The Captain has confirmed that any observable
FP on a noise-only slot is not acceptable. The original 6.0% tolerance was a calibration
convenience, not a product requirement. The revised target removes the tolerance entirely.

**Decision 2 — Calibrate in 0.05 increments starting at 0.20.**
Large threshold jumps (e.g. directly to 0.40) may skip the safe operating range: a threshold
just above the FP floor but well below the true-positive regression boundary. 0.05 steps
balance calibration speed against this risk. If the calibration loop reaches 0.35 without
finding 0 FPs, consult QA before continuing — the probability distribution may not be what
the R2 design assumed.

**Decision 3 — Separate result directories per calibration step (Lesson 14).**
Running a diagnostic at the same SHA deposits observations into the same versioned result
directory, causing anomalous mixed-K data. Directing each step to a distinct `--run-dir`
path avoids this. Only the final passing result directories are committed.

**Decision 4 — Single shim version increment (20260028) for the final committed threshold.**
Incrementing the shim version for each intermediate threshold step would produce a series of
stale shim versions with no meaningful ABI difference. One increment covers the final calibrated
binary.

**Decision 5 — NFR-021 fix is independent of threshold calibration.**
The callsign substitution affects no binary and no test logic. It can be committed before the
calibration loop runs, in the same PR, or in a separate commit on the branch. Either is
acceptable. What is not acceptable is merging to `main` with real callsigns in VCS.
