## 1. Context

### What R1 achieved

The R1 filter (commit `d40b4cd` on `fix/d009-fp-callsign-filter`) correctly eliminated the FP
categories observed in the initial diagnostic run (`6e821fa`):
- D9-R1 removed blank/whitespace messages (5+ events in `6e821fa`; 0 in `d40b4cd`).
- D9-R3 removed oversized-callsign messages (`CQ ETRHB0I3RYO`, `CQ GKC5JNL82FW`, etc.; 0 in `d40b4cd`).

These were real bugs; R1 was the right first response.

### Why R1 is insufficient

The R1 verification run produced **16 FP events across 7 of 12 trial slots = 133.33% event FP rate**
(threshold ≤ 6.0%).  Every FP in the run has a base callsign ≤ 6 characters — the exact D9-R3
guard limit.  QA analysis identified six gap categories; the full breakdown is in Section 5 of
`qa/rr-study/results/2026-06-20-d40b4cd/report.md`.  The key finding is:

**Text-level heuristics have a hard ceiling.** Even fixing all addressable text-level gaps (Gaps A
and C, which yield 7 event reductions), the residual minimum is ≥ 9 FP events / 12 slots = 75%
event FP rate — still an order of magnitude above threshold.  The root cause is the OSD CRC-14
false-alarm rate: at 529 trial codewords per candidate, pure noise produces enough CRC-14
coincidences that a subset of them decode to messages with structurally valid callsigns and
Maidenhead grids.  No text filter can distinguish these from real weak-signal QSOs.

### The required fix

A **native-layer OSD confidence gate** must be inserted in `decode.c` immediately after a
successful `osd_decode` call.  The gate computes the normalised correlation between the decoded
codeword and the input LLR array and rejects candidates below a threshold.

This is a change to `native/ft8_lib_build/patched/ft8/decode.c` plus the text-level extensions
described below.  No changes to the P/Invoke struct (`Ft8NativeResult`), no C# MVVM changes,
no UI changes.

### Action 1 — Native OSD correlation gate in `decode.c`

**File:** `native/ft8_lib_build/patched/ft8/decode.c`

There are two `osd_decode` call sites in the file (lines ~595 and ~753 as of shim 20260025).
Apply the gate at **both** sites.

#### 3.1a Correlation computation

Immediately after each `if (!osd_decode(llr_for_osd, 2, plain174))` block that returns `false`
on failure, add the following confidence check **before** the `status->ldpc_errors = 0;` line:

```c
if (!osd_decode(llr_for_osd, 2, plain174))
    return false;

/* OSD confidence gate (shim 20260026, D-009):
 * Reject candidates where the decoded codeword has low correlation with the
 * input LLRs.  For a genuine signal the decoded bits are predominantly aligned
 * with the LLR signs (score >> 0).  For a CRC-14 coincidence from pure noise
 * the bits are uncorrelated with the noise LLRs (score ≈ 0).
 *
 * corr = Σ hard_pm1[i] × llr[i]   (positive = decoded bit agrees with LLR sign)
 * norm = Σ |llr[i]|
 * score = corr / norm ∈ [−1, +1]
 * Threshold OSD_CORR_THRESHOLD: calibrated at 0.10; tune if S7 false-negatives appear.
 */
{
    float osd_corr = 0.0f;
    float osd_norm = 0.0f;
    for (int i = 0; i < FTX_LDPC_N; ++i) {
        float hard_pm1 = (plain174[i] == 0) ? 1.0f : -1.0f;
        osd_corr += hard_pm1 * llr_for_osd[i];
        osd_norm += fabsf(llr_for_osd[i]);
    }
    if (osd_norm > 0.0f && (osd_corr / osd_norm) < OSD_CORR_THRESHOLD)
        return false;
}

status->ldpc_errors = 0;
```

#### 3.1b Define the threshold constant

Add the following near the top of `decode.c`, after the existing `#include` block and before
the first function definition:

```c
/* OSD confidence gate threshold (shim 20260026, D-009).
 * Normalised correlation score below which an OSD-found codeword is rejected as
 * a CRC-14 false alarm.  Range [−1, +1]; initial value 0.10 (calibrated from
 * S5 FP analysis 2026-06-20).  Increase if S5 FP rate remains elevated; decrease
 * if S7 co-channel decode rate regresses. */
#define OSD_CORR_THRESHOLD 0.10f
```

#### 3.1c LLR sign convention note

In ft8_lib, `llr_for_osd[i] > 0` means bit i is more likely **0** (positive LLR = favours
the zero hypothesis).  `osd_decode` returns `plain174[i] = 0` or `1`.  The mapping above
(`hard_pm1 = (plain174[i] == 0) ? 1.0f : -1.0f`) therefore produces a **positive** product
when the decoded bit and the LLR agree (both point to bit=0 or both to bit=1), which is the
correct sign for correlation.

**Verify this on a known-good decode:** Add a temporary `printf("osd_corr_score: %.4f\n",
osd_corr / osd_norm);` inside the confidence block during development, decode a strong S4
fixture (SNR ≥ −10 dB, blind), and confirm the score is clearly above 0.10.  Remove the
`printf` before committing.

### Action 2 — Text-level extension: Gap A (2-token messages, both tokens)

**File:** `src/OpenWSFZ.Ft8/Ft8Decoder.cs` — `IsPlausibleMessage`

**Current code (D9-R3, 2-token branch):**
```csharp
if (firstSpace > 0 && secondSpace < 0)
{
    // 2-token message: "TOKEN0 TOKEN1"
    string token0 = text[..firstSpace];
    string token1 = text[(firstSpace + 1)..];
    if (token0 is "CQ" or "DE" or "QRZ" && IsCallsignOversized(token1))
        return false;
}
```

**Replace with:**
```csharp
if (firstSpace > 0 && secondSpace < 0)
{
    // 2-token message: "TOKEN0 TOKEN1"
    // Check both tokens — the CQ/DE/QRZ restriction on token0 was too narrow;
    // patterns like "<...> M5E5B91HFHL" and "9ULLPTCDZH <...>" slipped through.
    // IsCallsignOversized already exempts hash references (<...) and short keywords.
    string token0 = text[..firstSpace];
    string token1 = text[(firstSpace + 1)..];
    if (IsCallsignOversized(token0) || IsCallsignOversized(token1))
        return false;
}
```

This catches `<...> M5E5B91HFHL` (token1 = 11 chars), `<...> 7P8R9J2R3BG` (11 chars),
`<...> UF5NDNNJD2P` (11 chars), and `9ULLPTCDZH <...>` (token0 = 10 chars, base 10 > 6).

Valid messages must continue to pass — verify:
- `CQ Q1ABC` → token0 = `CQ` (2 chars, ≤ 3, exempt); token1 = `Q1ABC` (5 chars, not oversized) ✓
- `CQ <...>` → token0 exempt; token1 starts with `<`, exempt ✓
- `<...> Q1ABC` → token0 exempt; token1 = 5 chars, not oversized ✓
- `Q1ABC <...>` → token0 = 5 chars, not oversized; token1 exempt ✓
- `<...> RR73` → token0 exempt; token1 = `RR73` (4 chars, base 4 ≤ 6, total 4 ≤ 10, not oversized) ✓

### Action 3 — Text-level extension: Gap C (CQ 3-token grid validation)

**File:** `src/OpenWSFZ.Ft8/Ft8Decoder.cs` — `IsPlausibleMessage`

**Remove this line entirely:**
```csharp
// CQ messages (first token "CQ") are always valid.
if (text.StartsWith("CQ ", StringComparison.Ordinal)) return true;
```

**Rationale:** A valid 3-token CQ FT8 message is always `CQ CALLSIGN GRID`.  The third token
must be a Maidenhead grid locator (4 chars, letters in [A–R]).  The early return was bypassing
this check, allowing `CQ 3QQF EXLJSR` (`EXLJSR` = 6 chars, not a valid grid) and
`CQ /UX 6PY23BM` (`6PY23BM` = 7 chars, not a valid grid) through.

With the early return removed, CQ messages fall through to the existing grid/report validation:
- `CQ Q1ABC FN42` → last = `FN42`, valid grid (F, N both ≤ R) → **ACCEPT** ✓
- `CQ 3DA0MN KH51` → last = `KH51`, valid grid → **ACCEPT** ✓
- `CQ VK9AA OC12` → last = `OC12`, valid grid (O ≤ R) → **ACCEPT** ✓
- `CQ 3QQF EXLJSR` → last = `EXLJSR` (6 chars, not 4, not a dB report) → **REJECT** ✓
- `CQ /UX 6PY23BM` → last = `6PY23BM` (7 chars) → **REJECT** ✓

No change to the `IsDbReport` logic is needed; CQ messages with a dB report in the third
position are non-standard and are not expected in practice.

## 5. Decisions Made and Why

**Decision 1 — Native gate rather than text-level expansion.**
QA analysis (`qa/rr-study/results/2026-06-20-d40b4cd/report.md`, Section 5) showed that even
fixing all text-level addressable gaps leaves a minimum 75% event FP rate.  Gaps D (structurally
valid `CQ <...>` messages) and E (3-token messages with correctly-sized callsigns and valid grids)
are indistinguishable from real traffic by text alone.  The native correlation gate targets the
root cause — OSD CRC-14 false alarms — rather than symptoms.

**Decision 2 — Normalised inner product, not Hamming score.**
The kgoba/ft8_lib `osd_decode` function does not return a quality score; it returns 0 or 1.  We
compute the score in the call site rather than modifying `osd_decode`'s signature, to minimise
the change surface.  The normalised inner product (corr/norm) is dimensionless and robust to the
LLR scale set by `ftx_normalize_logl` (which scales all LLRs to variance 24 — the absolute
magnitudes are comparable across cycles, so unnormalised correlation would also work, but the
normalised form is easier to reason about and threshold-calibrate).

**Decision 3 — `OSD_CORR_THRESHOLD = 0.10` as a starting value.**
At 529 OSD trials × ~50 candidates per cycle in pure noise, the expected normalised correlation
for a CRC-14 coincidence is ≈ 0 (noise bits have no systematic alignment with the decoded codeword).
For a genuine signal at SNR ≈ −20 dB (OSD's operating range), the soft LLRs are weak but still
correlated with the true codeword bits; empirical correlation scores in the S7 co-channel scenario
should be meaningfully above 0.10.  The value is a reasonable first cut; calibration against both
failure modes is required (see Action 6).

**Decision 4 — Text-level extensions retained as defence-in-depth.**
Gap A and Gap C are fixable without native changes and add depth.  Even with the correlation gate
active, removing these FP categories at the text layer is cheap and reduces pressure on the gate
calibration margin.

**Decision 5 — No change to `Ft8NativeResult` struct.**
Surfacing the correlation score to the C# layer would require expanding the native struct from 48
to 52 bytes, updating `ExpectedNativeSizeBytes`, and adding plumbing through `Ft8LibInterop` and
`Ft8Decoder`.  The gate applied in C is simpler and sufficient for V1.  If diagnostic visibility
of individual OSD correlation scores is needed in a future investigation, adding a struct field can
be done in a separate change.
