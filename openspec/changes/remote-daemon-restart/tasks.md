- [x] 3.3 In `Program.cs`, near the existing `startupLogger.LogInformation("OpenWSFZ started on port {Port}.", port);` line (`ApplicationStarted.Register`, ~line 556), if `options.RelaunchedFromPid` is set, log at Information that this instance was started as a relaunch and name the PID it replaces (`daemon-host` spec: "Startup with the flag is logged")
- [x] 3.4 Wrap the final `await app.RunAsync();` call (or split into `StartAsync` + `WaitForShutdownAsync` if needed to isolate just the bind step) in a retry loop, active only when `options.RelaunchedFromPid` is set: on a bind failure (address already in use), wait 500 ms and retry, up to a total budget of 20000 ms (matching `PttWatchdog`'s existing constant, for consistency); log each retry at Debug. On budget exhaustion, log an Error and exit non-zero, matching the existing "abnormal exit uses non-zero code" `daemon-host` convention. When `RelaunchedFromPid` is absent, behaviour is completely unchanged (fail immediately on the first bind failure). **Implementation note:** literally retrying `app.StartAsync()` on the same instance is impossible — Kestrel marks the host "started" on the first call regardless of outcome, so a second call throws `InvalidOperationException` instead of retrying. `DaemonStartup.StartWithBindRetryAsync` instead retries a raw TCP bind probe on the configured port and calls the real `app.StartAsync()` exactly once, after the probe succeeds — see design.md Decision 4's correction note. Same observable behaviour for every scenario below.
- [x] 3.5 Tests proving: (a) a normal (non-relaunch) startup still fails fast on a port conflict — no behaviour change; (b) a relaunch-flagged startup retries and eventually succeeds once a competing listener on the same port is closed partway through the retry window (use a throwaway `TcpListener`/`HttpListener` bound to a test port to simulate "old instance still holding the port," closing it after a short delay); (c) a relaunch-flagged startup that never finds the port free exits non-zero once the budget is exhausted (use a short test-only budget override rather than waiting the full 20 s). Plus a regression guard confirming the real start step is invoked at most once regardless of probe-retry count.

## 4. Restart endpoint

- [x] 4.1 In `WebApp.cs`, alongside the existing `IQsoController`/`IPttController` service captures (~lines 808-815) and the `/api/v1/ptt/test` endpoint (~lines 987-1019), add `POST /api/v1/system/restart`. Check `qsoController?.Keying == true` and return `409 Conflict` ("a QSO is currently transmitting") exactly matching the `/api/v1/ptt/test` guard's structure and message style
- [x] 4.2 On success, respond `202 Accepted` with a small JSON body (e.g. `{ "status": "restarting" }`) immediately, then schedule the actual relaunch on a fire-and-forget `Task` after a short fixed delay (~500 ms) so the HTTP response has time to flush to the client first
- [x] 4.3 The scheduled task: log the resolved relaunch command (task 3.1) at Information; call `Process.Start` with it. If `Process.Start` itself throws (e.g. resolved executable not found), log the failure and stop — do **not** proceed to stop the current instance (the daemon must remain up and serving). If the spawn succeeds, call `app.Lifetime.StopApplication()`, which reuses the existing, unmodified `ApplicationStopping` shutdown chain (WebSocket abort → capture-pipeline stop → log flush → exit 0). **Implementation note:** the spawn+log logic lives behind a new `IDaemonRelauncher` seam (`OpenWSFZ.Abstractions`, implemented by `DaemonRelauncher` in the Daemon project, registered via `configureServices`) rather than inline in `WebApp.cs`, since `OpenWSFZ.Web` does not reference the Daemon assembly — mirrors the existing `ICatController` pattern exactly.
- [x] 4.4 Integration tests (mirroring `ConfigApiNullGuardTests.cs`'s `WebApplicationFactory`-based style): 409 when `IQsoController.Keying` is `true`; 202 when not keying (inject a fake/no-op relaunch mechanism via a test seam so the test process itself is never actually replaced — do not let this test spawn a real child process or call `StopApplication()` on the test host); 401 for a non-loopback request with a wrong/missing passphrase when `RemoteAccess` is configured with one, confirming the endpoint is covered by the existing blanket auth middleware with no bespoke wiring
