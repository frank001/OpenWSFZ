## 3. Platform detach primitives

- [x] 3.1 Add a small internal helper (e.g. `ConsoleDetacher` or similar) exposing a single `Detach()` method. On Windows, `[LibraryImport("kernel32.dll", SetLastError = true)] private static partial bool FreeConsole();` (design.md Decision 1 — `LibraryImport`, not `DllImport`, for AOT safety) wrapped as a partial method per the source-generator's requirements; call it and log the result (success/failure) at Debug. On Linux/macOS, `PosixSignalRegistration.Create(PosixSignal.SIGHUP, ctx => ctx.Cancel = true)`, retaining the registration for the process lifetime (e.g. a `static` field) so it is never garbage-collected/disposed prematurely
- [x] 3.2 Gate the platform branch at runtime with `OperatingSystem.IsWindows()` (design.md Context — `PosixSignal.SIGHUP` throws `PlatformNotSupportedException` on Windows; `FreeConsole` is a Windows-only API)
- [x] 3.3 Unit/integration tests: on Windows, confirm `FreeConsole()` P/Invoke does not throw when called from a test process (best available automated signal — genuinely proving "closing the console doesn't kill it" needs the manual acceptance gate below); on Linux/macOS, spawn a real child `Process` that registers the SIGHUP-ignore handler and blocks, send it a real `SIGHUP` (e.g. via `kill -HUP <pid>` or `Process.Kill` is not applicable — use a raw signal send, matching this project's existing `DaemonStartupTests`-style "spin up a real subprocess and observe it" integration-test convention), and confirm it is still running afterward; a control case without the handler confirms the same signal WOULD normally terminate it

## 5. Cold-start `--background`: spawn, then bounded confirmation

- [x] 5.1 When `options.Background` is true and `options.IsBackgroundWorker`/`options.RelaunchedFromPid` are both absent (an ordinary interactive cold start requesting detachment): resolve the relaunch command via the existing `DaemonRelaunch.ResolveCommand`-family mechanism (task 6.1's new parameter set to append `--background-worker` instead of/in addition to how `--relaunched-from` is appended), spawn it via the same `Process.Start` pattern `DaemonRelauncher` already uses, and — on success — poll `GET /api/v1/status` on the resolved port for up to 5 seconds at 500 ms intervals (design.md Decision 5) before exiting
- [x] 5.2 On confirmed startup within the budget: print a line naming the child's PID and the resolved log file path, exit 0. On budget exhaustion: print the "could not confirm yet, check `<logfile>`" caveat, exit 0 regardless (the spawn itself already succeeded)
- [x] 5.3 On the spawn itself failing (e.g. resolved executable not found): print a diagnosable error and exit non-zero — SHALL NOT report a successful background launch (mirrors `remote-daemon-restart`'s existing "spawn failure aborts before touching anything else" precedent)
- [x] 5.4 Tests: confirmed-within-budget path (fake/short-circuited status check); budget-exhausted path (short test-only budget override, not the real 5 s); spawn-failure path (non-existent resolved executable) exits non-zero without printing a false success

## 7. Observability: forced file logging, suppressed console sink

- [x] 7.1 `LoggingPipeline.Apply` (or its caller in `Program.cs`) gains a `suppressConsoleSink: bool` parameter, `true` when `options.IsBackgroundWorker`; when true, skip `.WriteTo.Console(...)` entirely rather than configuring it and leaving it to fail against invalid handles (design.md Decision 4) — **QA-found defect, fixed pre-merge**: the bootstrap `Apply()` call correctly passed this, but the separate `configStore.OnSaved` hot-reload call site (fires on every Settings-page save that touches `logLevel`/`logging`) did not, so a background worker's console sink could be silently reconfigured mid-life by an ordinary, unrelated config save — reachable in practice (e.g. an operator toggling `logging.fileEnabled` after reading the Warning line task 7.2 logs). Fixed by propagating `suppressConsoleSink: options.IsBackgroundWorker` at that second call site too (`Program.cs`, the `configStore.OnSaved` handler).
- [x] 7.2 When `options.IsBackgroundWorker` and the resolved `LoggingConfig.FileEnabled` is `false`, override it to `true` in memory only (never write back to `config.json`) before calling `Apply`; log one Warning-level line naming the resolved log file path before console-sink suppression would otherwise make that line unreachable (i.e. this line must itself go through a still-configured sink at the moment it's emitted — sequence this carefully relative to task 7.1)
- [x] 7.3 Tests: background-worker startup with `logging.fileEnabled: false` in the persisted config still produces a log file, and the persisted `config.json` itself remains unchanged after the process exits; background-worker startup never configures a console sink (assert via `LoggingPipeline`'s public surface, not by capturing real console output)

## 8. Acceptance Gate — manual verification (cross-platform, no ham-radio hardware required)

- [x] 8.1 **Windows**: start the daemon via `dotnet run -- --background` from an interactive terminal; confirm the shell prompt returns within ~5 s with a PID and log-file-path confirmation; confirm `Get-Process` (or Task Manager) shows the daemon running; **close the originating terminal window entirely**; confirm (via a new, unrelated terminal hitting `GET /api/v1/status`) the daemon is still running and serving — verified live by the Captain; caught and fixed two real defects along the way (arg-duplication across successive restarts; an orphaned throwaway orchestrator log file), both closed with regression tests (`DaemonRelaunchTests`'s "defect fix" block, `BackgroundColdStartE2ETests`)
- [x] 8.2 **Windows**: with the background instance from 8.1 still running, trigger `POST /api/v1/system/restart` from the Settings page; confirm the replacement instance is also running detached (repeat 8.1's terminal-close check against the new PID) — proving propagation across restarts (design.md's core requirement, directly requested by the Captain) — verified live by the Captain
- [x] 8.3 **Linux or macOS** (whichever is available; note which in the result): start the daemon via `dotnet run -- --background` from an interactive terminal (or a published apphost, matching `remote-daemon-restart`'s own 7.1/7.2 precedent of exercising both launch shapes if feasible); confirm the shell returns promptly; close the terminal (or send the terminal's controlling process a real `SIGHUP` if a full terminal-close isn't practical in the environment being used); confirm the daemon is still running and serving afterward — verified by QA on real Debian/WSL2 (genuine Linux kernel), **both** launch shapes: (1) framework-dependent `dotnet OpenWSFZ.Daemon.dll --background`, orchestrator exited in 753 ms, worker survived a real `kill -HUP`, `GET /api/v1/status` returned 200 before and after, forced-file-logging Warning line confirmed in the log file, persisted `config.json`'s `logging.fileEnabled` confirmed unchanged (`false`) on disk; (2) self-contained `PublishAot=true` apphost (`dotnet publish -r linux-x64`), same scenario, worker relaunched itself directly (no `dotnet` muxer prefix) — orchestrator exited in 566 ms, survived a real `SIGHUP`, still serving afterward. A negative control (identical process without `--background-worker`) confirmed the same real `SIGHUP` terminates it, proving the treatment case's survival wasn't a delivery fluke. macOS: not available in this environment — accepted by the Captain as untested.
- [x] 8.4 Confirm a **non-background** cold start and a **non-background** restart are both completely unaffected (existing `remote-daemon-restart` acceptance-gate scenarios 7.1/7.2 still hold) — this change must be provably inert when `--background`/`--background-worker` are absent — verified live by the Captain
