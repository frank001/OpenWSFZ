## Context

The R&R synthesiser (`qa/rr-study/`) is a pure-Python QA instrument. Its `synth.*`
package encodes FT8 messages to GFSK audio; `channel.py` adds AWGN and bandlimiting.
Every existing entry point (`synth_wav.py`, `run_study.py`, `resume_study.py`,
`harness/run_scenario.py`) requires a valid FT8 message text as input.

There is no facility to produce audio that does not originate from an FT8 encode:
no sine tones, no sweeps, no arbitrary noise bursts, no steady carriers. This blocks
several practical uses: interference scenario construction, waterfall calibration tones,
decoder stress tests with non-FT8 signals, and ad-hoc audio for perceptual QA.

The `synth.*` package itself contains all the numpy/scipy plumbing needed for AWGN;
the primitives for other signal types (sine, chirp, etc.) are straightforward numpy
one-liners. The gap is the absence of a render loop that treats signal generation as a
composable operation independent of FT8.

## Goals / Non-Goals

**Goals:**
- Provide `qa/rr-study/siggen.py` — a general-purpose scene renderer that mixes
  arbitrary signal types into a single output buffer and routes the result to a file,
  an audio device, or both.
- Support scene description via JSONL (one signal descriptor per line).
- Support all primitive signal types without any FT8 dependency: `sine`, `square`,
  `sawtooth`, `triangle`, `chirp`, `noise`.
- Support `ft8` as an optional signal type (lazy-imports `synth.encoder` only when
  needed); a scene with no `ft8` lines runs entirely without the FT8 encode chain.
- Support batch mode for unattended rendering of multiple scenes from a single JSON
  array file.
- Document all capabilities in `docs/siggen-reference.md` and cross-reference from
  the existing `docs/rr-synth-cli-guide.md`.
- Include evaluated recommendations for R&R scenario improvements enabled by `siggen`.

**Non-Goals:**
- Modifying any existing script (`synth_wav.py`, run/resume/harness family).
- Touching the .NET solution or any file outside `qa/rr-study/` and `docs/`.
- Introducing new Python package dependencies.
- Multi-channel (stereo) output — mono only, consistent with the rest of the toolchain.
- Real-time streaming or interactive mode.
- MIDI or scored music generation.

## Decisions

### D-1 — JSONL (JSON Lines) as the scene file format

Each line in the scene file is an independent JSON object — a single "parameter line"
describing one signal or the scene metadata. This matches the operator's mental model
("one instruction per line"), is trivially generated by scripts, is easy to append to,
and requires no JSON array wrapper. A line beginning with `#` is treated as a comment
and skipped (strict JSON does not support comments; we pre-process the line before
`json.loads`).

Alternatives considered:
- **JSON array object**: Self-contained but requires the operator to maintain bracket
  nesting; inserting a line in the middle of a large array is fragile.
- **CSV**: Familiar but structurally inadequate for optional/heterogeneous fields across
  signal types.
- **TOML / YAML**: Expressive but would require new Python package dependencies.

### D-2 — `"type":"scene"` line for output routing and global parameters

Output sinks (`out`, `device`) and global parameters (`sample_rate`, `duration_s`,
`seed`) are specified via a JSON object with `"type":"scene"` anywhere in the JSONL
file (conventionally the first line). If multiple `"type":"scene"` lines appear, the
last one encountered wins. CLI flags (`--out`, `--device`, `--rate`) always override
the file's scene parameters.

At least one of `out` or `device` must be resolvable (from file or CLI); the script
exits with a descriptive error otherwise.

This keeps the file self-describing — it can be run with no CLI flags when the scene
line specifies both output paths. Operators who prefer CLI-only can omit the scene line
entirely and supply all output parameters on the command line.

### D-3 — `amplitude` and `level_dbfs` are mutually exclusive; both are optional

All signal types accept either `amplitude` (linear peak, default 1.0) or `level_dbfs`
(0 dBFS = amplitude 1.0; −6 dBFS ≈ 0.5). Providing both in one signal object is an
error caught at parse time. When neither is provided, `amplitude` defaults to 1.0.

This avoids silent ambiguity while keeping the common case (just set `amplitude: 0.5`)
concise.

### D-4 — Signal rendering is independent; mixing is a simple numpy sum

Each signal type has a `render(descriptor, n_samples, sample_rate) → np.ndarray`
function that produces a full-length float64 array of `n_samples` zeros with the signal
placed at `start_s`. Mixing is the element-wise sum of all rendered arrays. No special
mixing logic (e.g., normalisation, bus routing) is applied before the final output step.

The output step writes the summed buffer via `wavio.write_wav()` (16-bit PCM, 6 dB
headroom normalisation) for file output, and/or normalises to 0.9 peak and passes the
float32 array to `sounddevice.play()` for device output — exactly the pattern already
established by `run_scenario.py`.

### D-5 — FT8 signal type uses lazy import of `synth.encoder`

When the parser encounters a line with `"type":"ft8"`, it imports `synth.encoder` at
that point. Scenes containing only primitive signal types never import the FT8 encode
chain. This keeps `siggen.py` usable even if `synth.encoder` is unavailable (e.g., on a
machine where the FT8 library is not set up), provided no `ft8` lines are present.

### D-6 — Batch mode uses a JSON array of scene objects

`--batch jobs.json` reads a JSON array where each element is a scene object: a dict
with `signals` (array of signal descriptor objects — same schema as JSONL lines without
the line-per-signal format), and optionally `out`, `device`, `sample_rate`,
`duration_s`. Batch items are processed sequentially; a failure in one item prints an
error and continues to the next (non-fatal batch semantics). CLI flags (`--out`,
`--device`, `--rate`) are applied as overrides to every batch item.

JSON Lines is not used for the batch file because each batch item is itself a
multi-signal object that does not decompose naturally into independent lines.

### D-7 — Total scene duration auto-computed if not specified

If `duration_s` is absent from the `"type":"scene"` line (and not supplied on the CLI),
the total buffer length is `max(start_s + duration_s)` across all signal descriptors.
For `ft8` type signals, which have no explicit `duration_s` (fixed by the FT8 protocol
at `SLOT_LENGTH_S = 15.0`), the contribution is `start_s + 15.0`. This means a scene
with only signal lines and no `"type":"scene"` line auto-sizes its buffer correctly.

## Risks / Trade-offs

- **[Risk] Peak clipping on summed output** — summing many high-amplitude signals can
  exceed ±1.0 before `wavio.write_wav`'s normalisation step.
  → **Mitigation**: `wavio.write_wav` already peak-normalises with 6 dB headroom.
  Device playback path normalises to 0.9 peak (matching `run_scenario.py`). No
  additional action needed; document the normalisation behaviour clearly.

- **[Risk] `"type":"scene"` line position ambiguity** — operators may place the scene
  line in the middle of the file.
  → **Mitigation**: Last-wins semantics; first pass reads all lines to collect the scene
  config before the second pass renders signals. Document the convention clearly.

- **[Risk] Comment line pre-processing** — stripping `#`-prefixed lines before
  `json.loads` is a non-standard extension to JSON.
  → **Mitigation**: Clearly document it as a `siggen.py`-specific pre-processing rule,
  not a general JSON extension. Only single-line `#`-comments at line start are
  stripped; inline `#` within JSON strings is not affected.

- **[Risk] `ft8` signal `duration_s` auto-detection** — the auto-duration logic must
  know FT8 signals are 15 s wide without the operator specifying it.
  → **Mitigation**: The FT8 slot length is a named constant (`SLOT_LENGTH_S`); the
  auto-duration logic imports it from `synth.constants` only if an `ft8` line is
  present (consistent with D-5 lazy import).

## R&R Scenario Improvement Recommendations

The following improvements are enabled by `siggen.py`. They are recommendations for a
future study run — none are required before implementing `siggen.py` itself.

**R-001 — Revert S1/S2/S3 to wideband AWGN (HIGH priority)**
The bandlimited 3 kHz noise applied at commit `c9556a2` was correct for the perceptual
quality of multi-signal scenarios (S4, S7, S8) but broke S1's bias measurement: WSJT-X
bias shifted from +1.00 dB to −2.00 dB under bandlimited noise (the reference appraiser
now fails its own criterion), and %GR&R deteriorated from 1.40% (PASS) to 10.19%
(MARGINAL). S1 is a calibration measurement scenario; wideband AWGN is the correct
reference. Recommendation: restore `noise_cutoff_hz=None` in `_render_single()` for
S1/S2/S3 in `harness/run_scenario.py`.

**R-002 — S1b threshold sweep (MEDIUM priority)**
S1b currently tests one threshold at −21 dB (OpenWSFZ: 0%, WSJT-X: 100%), which gives
a cliff with no curve. A sweep from −24 to −18 dB in 1 dB steps characterises the
sensitivity boundary precisely for both appraisers. With `siggen.py` batch mode, the
operator can generate all threshold WAVs for manual waterfall inspection in one command
before formalising the sweep as a scenario JSON.

**R-003 — S7 near-collision sub-scenario (MEDIUM priority)**
S8 stations E/F (12 Hz apart) exercise frequency resolution, but S7 has no near-collision
part. `siggen.py` enables rapid construction of near-collision test audio:
```
{"type":"ft8","message":"Q1AW Q1ABC +05","freq_hz":1150,"amplitude":0.5,"start_s":0.0}
{"type":"ft8","message":"Q1ABC Q1AW RR73","freq_hz":1162,"amplitude":0.5,"start_s":0.0}
{"type":"noise","level_dbfs":-20,"cutoff_hz":3000,"start_s":0.0,"duration_s":15.0}
```
This can be trialled via `siggen.py` before adding a formal part to `s7-compounding.json`.

**R-004 — resume_study.py log path parameterisation (LOW priority)**
`WSJT_ALL_TXT` and `OWSFZ_ALL_TXT` in `resume_study.py` are hardcoded machine-specific
paths (F-002 from the `synth-cli-args` review). Adding `--wsjt-log` and `--owsfz-log`
flags would make the script portable to any machine. Low effort, high portability gain.
This is outside the scope of `siggen.py` but should be captured for the next
synthesiser CLI change.
