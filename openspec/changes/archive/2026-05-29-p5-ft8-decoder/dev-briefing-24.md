# Developer Briefing — p5-ft8-decoder (Round 24)

**Date:** 2026-05-27  
**Issued by:** QA  
**Branch:** `feat/p7-p6-logging-and-display`  
**Scope:** D10 — Costas soft-match fires trivially for real flat-spectrum noise

---

## Situation

All 48 tests pass on a fresh build. The synthetic fixture test
(`DecodeAsync_WavFixture_ReturnsKnownDecodes`) decodes in ~721 ms. No
regression in the test suite.

Despite this, the live application produces **no decodes** against real audio.
The log files in `logs/` tell the story clearly:

```
00:31:02  Cycle 22:30:00: 0 decode(s) found.   ← 62 s to decode a 15-s window
00:31:48  Cycle 22:31:00: 0 decode(s) found.   ← 46 s
00:32:32  Cycle 22:31:45: 0 decode(s) found.   ← 47 s
00:33:16  Cycle 22:32:30: 0 decode(s) found.   ← 44 s
00:34:01  Cycle 22:33:15: 0 decode(s) found.   ← 31 s
```

Each decode takes 30–62 seconds. The `framerOutput` channel has capacity 2
with `DropOldest`; at these decode times, at best every 4th cycle is
processed. The rest are silently discarded.

---

## Root Cause — D10

**File:** `src/OpenWSFZ.Ft8/Dsp/CostasSynchroniser.cs`, line 102  

The Costas scoring criterion is:

```csharp
if (maxE < -18f) continue;           // D9 gate
score += costas >= maxE - 0.1f ? 1.0f : 0.0f;
```

### Why D9 does not help for real audio

The D9 gate (`maxE < -18f`) was designed for the **synthetic-signal case** only.
When the PCM is generated by `TestFt8Encoder`, silent frequency bands have
**exactly zero power** in every FFT bin, giving `log(0 + 1e-10) = −23`. The
gate skips those bands.

For real audio captured from any hardware device, every FFT bin has non-zero
power — even at a −90 dBFS quantisation noise floor, `log(power) ≈ −7`. The
D9 gate **never fires** for real audio. All 24 123 Costas evaluations proceed
to the soft-match check.

### Why the soft-match is lethal on real audio

The criterion `costas >= maxE - 0.1f` with a tolerance of `0.1` natural-log
units corresponds to a linear-power ratio of `e^0.1 ≈ 1.105` — about 10%. In
a real audio noise band, all 8 FFT bins over a 50 Hz window have nearly equal
energy (flat-spectrum pink or white noise). Every tone trivially satisfies the
criterion:

| All 8 bins near energy E | costas ≥ maxE − 0.1? |
|---|---|
| Flat noise, bins ∈ [E−0.05, E] | Yes, for all 8 tones |
| Result: Costas score | 21/21 = **1.0** |

Every one of the 51 time-offsets × 473 frequency steps = **24 123** evaluations
scores above the 0.45 threshold and triggers a Goertzel computation
(79 × 15 × 1 920 = 2.3M operations). Total arithmetic per cycle: ~55 billion
operations — consistent with the 30–62 seconds observed.

---

## The Fix

### Task 1 — Change soft-match to hard decision

**File:** `src/OpenWSFZ.Ft8/Dsp/CostasSynchroniser.cs`

```csharp
// BEFORE:
score += costas >= maxE - 0.1f ? 1.0f : 0.0f;

// AFTER:
score += costas >= maxE ? 1.0f : 0.0f;
```

That is the entire code change required.

**Why this is correct:**

For **flat noise** (all 8 bins ≈ equal): probability that the Costas tone is
the peak is 1/8 per symbol → expected score = 21 × 1/8 = 2.625 / 21 ≈ **0.125**,
well below the 0.45 threshold. No false positives. No Goertzel calls triggered.
Decode time returns to < 1 s.

For a **real FT8 signal at ≥ −15 dB SNR**: the correct tone carries most of
the symbol energy and is the peak for the majority of Costas symbols → score
> 0.45 → candidate passes to Goertzel correctly.

For **clean synthetic signals** (all existing tests): the correct tone has
overwhelmingly the highest energy. All existing tests continue to pass without
modification.

> **Note on the D9 gate:** The `if (maxE < -18f) continue` line can be
> retained — it still prevents the decoder from spending Costas-scoring CPU
> on genuinely empty spectrogram rows (e.g. rows zeroed by the D5 stale-row
> fix when `startSample` is near the end of the buffer). It is now redundant
> for the false-positive problem, but it is not harmful.

## After Applying the Fix

### Step 1 — Run the full test suite

```
dotnet test tests/OpenWSFZ.Ft8.Tests/ -c Release
```

Expected: 48 passed, 0 failed.

### Step 2 — Confirm decode time

Run the daemon against the USB Audio CODEC (or any non-silent input). The log
should show:

```
Cycle HH:MM:SS: N decode(s) found.
```

appearing approximately **15 seconds** after each cycle boundary, not 30–60
seconds. If the decode message arrives within 5 seconds of the cycle boundary,
D10 is resolved.

### Step 3 — Confirm live decodes

Use an audio source that is **known to contain FT8 signals**. Options:

1. **WAV playback**: download a real FT8 recording from
   `https://www.sigidwiki.com/wiki/FT8` or from the WSJT-X project's test
   vectors. Play it through Voicemeeter (routed to the B2 output bus).
   WSJT-X decodes from the same file should match OpenWSFZ decodes.

2. **Live radio**: confirm a band with active FT8 (40 m / 7.074 MHz is
   reliable most hours; use a site such as `pskreporter.info` or
   `wsprnet.org` to verify activity before testing).

3. **WSJT-X cross-check**: run WSJT-X and OpenWSFZ on the same audio device
   simultaneously. Any message WSJT-X decodes that OpenWSFZ does not is a
   remaining defect.

The application is not ready to merge until at least one live or WAV-playback
decode is confirmed and logged.
