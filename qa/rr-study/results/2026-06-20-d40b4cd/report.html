Section 1 — Study Hypothesis

Purpose of this run: Targeted S5-only verification that the D-009 R1 false-positive filter (commit d40b4cd) reduces the OSD-induced S5 FP rate below the gate threshold of 6.0%.

Background: The OSD fallback introduced at shim 20260025 (fix/d001-osd-fallback, merged 1809ce7) evaluates up to 529 trial codewords per candidate when bp_decode fails to converge. In pure noise (S5) or a busy band (S8) this generates CRC-14 coincidences at a non-trivial rate, producing garbage messages in ALL.TXT and the UI. The full S1–S8 gate run at 6e821fa (2026-06-20) measured S5 FP rate = 91.67% — gate-blocking FAIL. That run raised defect D-009.

D-009 R1 fix (commit d40b4cd): Three new guard rules added to Ft8Decoder.IsPlausibleMessage: - D9-R1 — reject blank / whitespace messages. - D9-R2 — reject single-token strings ≥ 16 chars of pure uppercase hex (unrecognised message type rendered as hex dump). - D9-R3 — reject any callsign-position token whose base callsign exceeds 6 chars or whose total length (including /suffix) exceeds 10 chars.

Null hypothesis under test:

  • H₀_D9: D9-R1 + D9-R2 + D9-R3 together reduce the S5 FP rate to ≤ 6.0%. Rejection criterion: S5 FP rate (OpenWSFZ) > 6.0% on the same 12-slot S5 noise-only protocol used in 6e821fa.

What constitutes a meaningful result:

  • H₀_D9 NOT REJECTED (PASS, FP rate ≤ 6.0%): Filter is sufficient; branch ready for merge review.
  • H₀_D9 REJECTED (FAIL, FP rate > 6.0%): Filter is insufficient; D-009 requires a further revision before merge.

Defect IDs under observation: D-009 (OSD FP filter — first implementation attempt).

Section 5 — Recommendations

D-009 (OSD false-positive filter) — filter R1 is insufficient. Branch fix/d009-fp-callsign-filter CANNOT MERGE.

The R1 filter (D9-R1/R2/R3) eliminated the specific FP categories observed in the 6e821fa diagnostic run — blank messages and oversized callsigns (>6-char base, e.g. ETRHB0I3RYO, GKC5JNL82FW). However, the d40b4cd verification run reveals six distinct categories of FP that the filter does not address. Every FP in this run has a base callsign ≤ 6 characters — the exact guard boundary of D9-R3.

Text-level ceiling

Even if Gaps A and C are fixed (7 events removed), the remaining Gaps B, D, E, F leave ≥ 9 FP events across ≥ 6 of the 12 trial slots — a minimum of 75% event FP rate. Text-level heuristics cannot bring the OpenWSFZ S5 FP rate below the 6% gate threshold. The OSD CRC-14 false-alarm rate at 529 trials per candidate is simply too high; a subset of noise coincidences always decode to structurally valid-looking FT8 messages.

Required fix — native OSD confidence gate (D-009 R2)

After osd_decode finds a CRC-14-valid codeword, compute the normalised correlation between the decoded codeword and the input LLR array:

corr = Σ (hard_pm1[i] × llr[i]) for i = 0..173
       where hard_pm1[i] = +1 if plain174[i] == 0, −1 if plain174[i] == 1
norm = Σ |llr[i]|
score = corr / norm   ∈ [−1, +1]

For a genuine signal at any SNR where OSD can recover the message, the decoded bits are predominantly aligned with the soft LLR signs → score ≫ 0 (expected 0.2–0.5 at decodable SNR). For a CRC-14 coincidence from pure noise, the 174 decoded bits are uncorrelated with the noise LLRs → score ≈ 0.

Reject the OSD result if score < OSD_CORR_THRESHOLD. Initial recommended threshold: 0.10 (10% normalised correlation). This value requires calibration against S5 (reject) and S7 P0 (accept) empirical distributions; the handoff document specifies the calibration procedure.

Text-level extensions (still required regardless)

Even with the correlation gate in place, Gaps A and C should be fixed as additional defence-in-depth:

  • Gap A fix: Extend D9-R3 2-token branch to apply IsCallsignOversized to both tokens, not only to token1 when token0 is CQ/DE/QRZ. (Hash references are already exempt via the StartsWith('<') guard in IsCallsignOversized.)
  • Gap C fix: Remove the if (text.StartsWith("CQ ", StringComparison.Ordinal)) return true; early-return from IsPlausibleMessage. With this removed, CQ 3-token messages are subject to the same grid/report validation as Standard QSO messages, which correctly rejects CQ 3QQF EXLJSR (6-char third token, not a valid grid) while accepting CQ Q1ABC FN42 (valid 4-char grid).
