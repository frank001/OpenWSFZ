# R&R-007 — Attribute-Kappa Ratification Investigation (S4 unit-of-analysis defect)

**Raised:** 2026-07-07
**Raised by:** QA
**Status:** Investigation complete — recommendation pending Captain decision
**Triggered by:** Captain-directed investigation into whether the §9.3/§10 pooled
attribute-κ gate (currently advisory) is ready for ratification, following the
R&R-005 citation cleanup (GitHub #35, closed).
**Scope note:** This is a *different* question from R&R-005. R&R-005 was a
dangling documentation citation with no analytical content. R&R-007 is the
substantive investigation that citation was supposed to be tracking.

---

## 1. What was asked

STUDY-SPEC.md §9.3 conditions ratification of the pooled attribute-κ gate
(S4 injected messages = positives, S5 signal-free slots = negatives) on two
things:

1. Captain ratification of the **pooled method** itself.
2. Restricting the **positive population to decodable SNRs** (cf. the R&R-005
   S1 redesign — excluding SNRs below the established −12 dB reliable-decode
   floor) "so κ measures agreement rather than decode probability."

The brief was to investigate whether these conditions can now be satisfied.

---

## 2. What the investigation found

### 2.1 Confirmed: condition 2's concern is real, but is not the binding constraint

S4's positive population (`scenarios/s4-density.json`) spans SNRs from +3 dB
down to −24 dB — the same sub-threshold range R&R-005 excluded from S1:

| Part | n_signals | Sub-threshold (< −12 dB) signals |
|---|---|---|
| 0 | 1 | 0 |
| 1 | 5 | 0 |
| 2 | 10 | 1 |
| 3 | 20 | 8 |
| 4 | 30 | 12 |
| **Total** | **66** | **21 (31.8%)** |

Just as R&R-005 found for S1's ANOVA, decode misses on these 21 signals are a
decode-capability boundary, not a measurement disagreement — exactly the
contamination §9.3 flagged. **However**, investigating this turned up a more
fundamental problem that makes the SNR-restriction question currently
unanswerable from existing data (see 2.2).

### 2.2 New finding: S4's matcher cannot measure per-message recovery at all

`harness/matcher.py::_match_appraiser` (lines 148–195) and the S4 truth
generator record **one truth row per (part, trial)** — not one per injected
message. For multi-signal parts, `message_text` is a single `"; "`-joined pool
string (confirmed directly in `truth.csv`, e.g. part 4 trial 0: one row, 30
messages concatenated). The matcher's own comment states the consequence
explicitly:

> "S4: truth_msg is a `; `-joined pool of individual messages. A candidate
> matches if its decoded text equals **any one** pool entry."

The loop `break`s on the first pool hit and only **one** candidate decode is
ever consumed as the match for that cycle. Every other genuinely correct
decode of the remaining messages in that same busy cycle is left unconsumed
and falls through to Pass 2, where it is written out as a **false positive**
(854 such rows in the `793a298` run's `S4_matched.csv` — vastly more than
genuine spurious noise decodes could account for). These rows are excluded
from the pooled-κ population via the `false_positive == False` filter in
`_attribute_agreement`, so they don't corrupt the confusion matrix directly —
but it means **the fate of every message beyond the first successful match in
a cycle is simply discarded from every metric.** Nothing currently measures
"how many of the 30 injected messages did the appraiser actually recover."

### 2.3 The ceiling effect is visible in real data, and it invalidates the metric

Run `793a298` (`report.md`, "Attribute Agreement Analysis" section):

| Appraiser | TP | FN | FP | TN | Recovery | κ vs truth |
|---|---|---|---|---|---|---|
| WSJT-X | 15 | 0 | 0 | 12 | 100.00% | 1.000 |
| OpenWSFZ | 15 | 0 | 0 | 12 | 100.00% | 1.000 |

**Both appraisers show a perfect 15/15, κ = 1.000, PASS.** Every S4 part
includes at least one easy, high-SNR message (every part's `snr_db_set`
includes 0 dB or better), so "at least one of N matches" is nearly guaranteed
regardless of how the appraiser handles the other N−1 messages — the metric
saturates at ceiling and cannot distinguish good QRM handling from bad.

This is not a hypothetical concern. **The same run, same build**, reports S7
(which *does* track and match truth per individual message — S7 is not
susceptible to this defect) showing OpenWSFZ recovering only **40.00%** of
co-channel stacked messages against WSJT-X's **100.00%** (`report.md` §S7,
`co_channel` row). S4 and S7 are testing conceptually the same underlying
capability — decode performance when multiple signals compete — on the same
build, in the same run. One reports a real, substantial, well-known gap. The
other reports flawless perfection. The discrepancy is entirely explained by
S4's any-one-of-pool matching; it is not evidence that OpenWSFZ's QRM handling
is actually fine in S4's conditions.

---

## 3. Conclusion

**Ratification cannot proceed yet, and not for the reason originally
anticipated.** Restricting the positive population to decodable SNRs (§9.3
condition 2) would not fix the pooled-κ gate on its own — even a perfectly
SNR-filtered population would still be scored at the wrong unit of analysis
(per-cycle "any match" instead of per-message match), and would still ceiling
out at or near 1.000 for any appraiser that reliably decodes at least the
easiest signal per cycle. The §9.3 conditions, as written, are necessary but
not sufficient; a third, previously undocumented condition exists:

**Condition 3 (new): S4's truth generation and matcher must record and score
one truth row per injected message, not one pooled row per cycle**, before
either κ's pooled method or its SNR-restricted population can be meaningfully
evaluated — let alone ratified as a hard gate.

This is the same *class* of finding as R&R-005 (a scenario/harness design flaw
masquerading as a metric problem), but affects the matcher's core matching
logic rather than a scenario's parameter ladder, and is more consequential:
R&R-005's S1 ANOVA produced a visibly bad (FAIL) result that could not be
ignored. This defect produces a visibly *perfect* result that invites exactly
the wrong conclusion — that S4 is fine — while providing zero actual
discriminating power.

---

## 4. Recommendation

1. **Do not ratify the pooled attribute-κ gate at this time.** It remains
   advisory, correctly, and for a stronger reason than previously documented.
2. **Scope a harness redesign** (candidate acceptance criteria below) so S4
   records truth and match/miss status per individual injected message,
   carrying that message's own `true_snr_db`. This is what would make
   condition 2 (decodable-SNR restriction) actually computable, and would let
   S4 measure what its own description promises ("recovery Kappa... between
   appraisers") instead of "did anything decode."
3. Once redesigned, S4 should be **re-run** and the pooled-κ analysis
   re-evaluated against real per-message, SNR-aware data before any
   ratification decision is revisited.

### Candidate acceptance criteria for the redesign (draft, not yet ratified)

| Criterion | Required |
|---|---|
| S4 truth.csv | One row per (part, trial, message) — not one pooled row per cycle |
| S4 truth row | Carries that individual message's own `true_snr_db` |
| Matcher | Consumes at most one candidate per truth row; each message scored independently |
| S4_matched.csv | Row count per part = `n_signals × trials` (e.g. part 4 = 90 rows/appraiser, not 3) |
| Pooled κ (post-redesign) | No longer structurally guaranteed to ceiling at 1.000 for dense parts |
| Regression check | S7's existing per-message matching (already correct) is unaffected |
| `analyse.py` / `matcher.py` unit tests | Updated for the new S4 row shape; all pass |

## 5. Out of scope

- **Actually implementing the redesign** — this document is the investigation
  and recommendation only, per the Captain's request to "scope and start the
  investigation." Implementation is a separate, larger piece of harness work
  and should be its own tracked change once the Captain decides whether to
  proceed.
- **S4's FP-row leakage into other report sections** — the 854 mislabelled
  "false positive" rows are excluded from the current pooled-κ computation
  (verified: `_attribute_agreement` filters `false_positive == False`), so they
  do not corrupt today's numbers. Whether they leak into any *other* report
  section as misleading FP counts was not exhaustively checked across every
  report artifact and would be worth a quick pass if the redesign goes ahead.
- **Re-deciding the pooled method itself** (§9.3 condition 1) — not blocked on
  anything new found here; remains a separate Captain call.
