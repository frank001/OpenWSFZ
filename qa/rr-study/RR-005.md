## 1. Problem statement

Run `4c34ef6` produced a severe S1 regression relative to the previous run (`aa053a9`):

| Metric | `aa053a9` | `4c34ef6` | Change |
|---|---|---|---|
| S1 Repeatability σ² | 0.12 | **0.00** | Clamped |
| S1 Reproducibility σ² | 3.63 | **26.10** | ×7.2 |
| S1 %GR&R | 5.9% | **32.0%** | FAIL |
| S1 ndc | 5 | **2** | MARGINAL |
| WSJT-X decode miss rate | — | 23.3% (7/30) | — |
| OpenWSFZ decode miss rate | — | 30.0% (9/30) | — |

The S1 scenario had not changed between runs.  The regression is entirely attributable
to **threshold misses at low SNR creating an unbalanced ANOVA dataset**.

---

## 2. Root cause

### 2.1 Threshold misses invalidate the balanced ANOVA structure

The current S1 ladder spans {−24, −21, −18, −15, −12, …, +3} dB.  At the four
lowest levels (−24, −21, −18, −15 dB) neither app decodes reliably — the probability
of a successful decode per trial is well below 100%.

In a properly balanced two-way Gage R&R ANOVA:

```
SS_total = SS_part + SS_appraiser + SS_appraiser×part + SS_error
```

`SS_error` (within-cell) is estimated from the variance **within** each (part × appraiser)
cell across trials.  When a trial is a miss there is no measurement for that cell entry.
This produces two failure modes simultaneously:

1. **Different apps miss different cells.** WSJT-X missed 7 of 30 trials, OpenWSFZ
   missed 9 — and crucially, not the same 9.  When app A decodes trial (part=1, trial=2)
   but app B does not, the apparent difference in that cell is a decode threshold effect,
   not a measurement error.  This inflates the App×Part interaction term — captured
   under Reproducibility — with a capability boundary rather than noise.

2. **Within-cell variance goes negative.** With so few matched rows per cell (sometimes
   0 or 1 out of 3), the expected value of `MS_error` falls below `MS_appraiser×part` in
   the ANOVA algebra, yielding a negative variance estimate.  Per AIAG convention this is
   clamped to zero, which explains why Repeatability = 0.00 σ² — and why the entire GR&R
   budget collapsed into Reproducibility (26.10 σ²).

### 2.2 The problem was known and documented

STUDY-SPEC.md §16 already identified this root cause after run `46d7f6a` and prescribed
the fix.  This change raises it formally as R&R-005 so it is tracked and implemented.

## 3. Required changes

### 3.1 `scenarios/s1-snr-ladder.json` — restrict to reliable decode range

Replace the 10 parts with a new ladder that stays entirely above the decode threshold.
The requirement is: **both apps must decode ≥ 95% of trials at every part**.  Run data
confirms -12 dB is the reliable floor for both apps.

**Prescribed design:** 10 parts at 3 dB spacing extending upward from -12 dB:

```json
"parts": [
  {"part_index": 0, "snr_db": -12},
  {"part_index": 1, "snr_db":  -9},
  {"part_index": 2, "snr_db":  -6},
  {"part_index": 3, "snr_db":  -3},
  {"part_index": 4, "snr_db":   0},
  {"part_index": 5, "snr_db":   3},
  {"part_index": 6, "snr_db":   6},
  {"part_index": 7, "snr_db":   9},
  {"part_index": 8, "snr_db":  12},
  {"part_index": 9, "snr_db":  15}
]
```

Update `_comment` and `description` to record the redesign date and reference R&R-005.

> **Note on upper range:** SNR ≥ +9 dB parts will show near-zero within-cell variance
> (both apps decode perfectly, always reporting the same SNR for the same signal).
> This is acceptable — zero-variance cells do not break the ANOVA the way missing-cell
> imbalance does; they simply contribute zero to SS_error, which is handled correctly
> by the ANOVA method.  The wide range remains valuable for the Bias & Linearity chart.

### 3.2 `scenarios/s1b-snr-threshold.json` — new companion attribute scenario

Create a new file measuring **decode rate** (attribute: decoded / not decoded) at the
four SNR levels excluded from the redesigned S1.  This cleanly separates "how accurately
does the app measure SNR?" (S1) from "at what SNR does the app stop decoding?" (S1b).

Minimum required fields:

```json
{
  "_comment": "S1b Low-SNR Threshold — decode-rate study for SNRs excluded from the redesigned S1 ladder (R&R-005).",
  "_format_version": "1",

  "id": "S1b",
  "name": "Low-SNR threshold",
  "response": "decode_rate",
  "description": "4 parts × 2 appraisers × 3 trials. ...",

  "message_ids": ["MSG-01"],
  "fixed": {"base_freq_hz": 1500, "dt_s": 0.2},

  "analysis": "attribute_decode_rate",

  "parts": [
    {"part_index": 0, "snr_db": -24},
    {"part_index": 1, "snr_db": -21},
    {"part_index": 2, "snr_db": -18},
    {"part_index": 3, "snr_db": -15}
  ],

  "trials": 3,
  "seed_formula": "hash('S1b', part_index, trial_index)"
}
```

Fill out the `description` field as done for S3b — cite R&R-005 and note the companion
relationship to S1.

### 3.3 `harness/analyse.py` — generalise decode-rate analysis for S1b

The existing `_analyse_decode_rate()` and `_decode_rate_report_lines()` functions are
written specifically for S3b (DT is the part parameter, x-axis label is "True DT (s)",
and a vertical reference line is drawn at DT = 0).  S1b uses SNR as the part parameter.

**Three targeted changes are required:**

#### a. Expand `DECODE_RATE_SCENARIOS` and add a config dict

```python
# line ~42 — add "S1b"
DECODE_RATE_SCENARIOS = {"S3b", "S1b"}

# new constant (insert immediately below)
DECODE_RATE_CONFIG: dict[str, dict] = {
    "S3b": {
        "part_var":    "true_dt_s",
        "part_label":  "True DT (s)",
        "section_title": "Negative-DT decode boundary",
        "section_intro": (
            "Decode rate (% of injected messages recovered) as DT sweeps from 0.0 s "
            "down to -2.7 s.  Companion to S3; separates 'does it decode?' from "
            "'does it report DT accurately?'.  Informational - no AIAG threshold."
        ),
        "chart_ref_line": 0.0,   # vertical line at DT = 0
    },
    "S1b": {
        "part_var":    "true_snr_db",
        "part_label":  "True SNR (dB)",
        "section_title": "Low-SNR threshold study",
        "section_intro": (
            "Decode rate (% of injected messages recovered) at SNRs excluded from the "
            "redesigned S1 ladder (-24 to -15 dB).  Companion to S1; separates "
            "'does it decode at this SNR?' from 'how accurately does it measure SNR?'.  "
            "Informational - no AIAG threshold."
        ),
        "chart_ref_line": None,  # no reference line for SNR axis
    },
}
```

#### b. Generalise `_analyse_decode_rate()`

Add `part_var: str = "true_dt_s"` and `part_label: str = "True DT (s)"` parameters.
Everywhere the function currently uses the literal string `"true_dt_s"` as a column
name, replace it with `part_var`.  Replace the hardcoded `"true_dt_s"` key in the
`per_part` row dict with `"part_val"` so `_decode_rate_report_lines()` can use it
generically.

In the chart:
- Replace `ax.set_xlabel("True DT (s)")` with `ax.set_xlabel(part_label)`.
- Replace the unconditional `ax.axvline(0.0, ...)` with a conditional:
  `if cfg.get("chart_ref_line") is not None: ax.axvline(cfg["chart_ref_line"], ...)`.
  (Pass the `cfg` dict in, or read it from `DECODE_RATE_CONFIG[scen_id]`.)

Add `"part_var"` and `"part_label"` to the returned results dict so the rendering
function can use them without an extra parameter.

#### c. Generalise `_decode_rate_report_lines()`

- Read `part_label` from `results["part_label"]`.
- Read section title and intro from `DECODE_RATE_CONFIG[results["scenario_id"]]`.
- Replace `r['true_dt_s']` in the table body with `r['part_val']`.
- Replace the hardcoded S3b column header `"True DT (s)"` with `part_label`.

#### d. Update the dispatch loop in `main()`

The current dispatch finds only the **first** matching decode-rate scenario:

```python
# CURRENT (handles only one scenario)
if any(sid in matched for sid in DECODE_RATE_SCENARIOS):
    scen_id = next(sid for sid in DECODE_RATE_SCENARIOS if sid in matched)
    s3b_results = _analyse_decode_rate(matched[scen_id], scen_id, run_dir)
```

Replace with a loop that collects all matching scenarios:

```python
# REPLACEMENT (handles any number of decode-rate scenarios)
decode_rate_results: list[dict] = []
for sid in sorted(DECODE_RATE_SCENARIOS):
    if sid not in matched:
        continue
    cfg = DECODE_RATE_CONFIG[sid]
    dr = _analyse_decode_rate(
        matched[sid], sid, run_dir,
        part_var=cfg["part_var"],
        part_label=cfg["part_label"],
    )
    decode_rate_results.append(dr)
    for appr in APPRAISERS:
        rate = dr["overall"].get(appr, float("nan"))
        print(f"  {sid} decode rate ({appr}): {_fmt_num(rate)}%  (informational)")
```

Update the report-generation block accordingly:

```python
# CURRENT
if s3b_results:
    lines += _decode_rate_report_lines(s3b_results)

# REPLACEMENT
for dr in decode_rate_results:
    lines += _decode_rate_report_lines(dr)
```
